# orval-openapi

Use this skill when working with an Orval-generated API client: after a backend API changes, when editing `orval.config.ts` or the fetch mutator, or when consuming generated React Query hooks and types. The goal: generated code is regenerated, never hand-edited, every request goes through the shared mutator, and types are derived from the schema rather than re-declared.

## Triggers

- The backend API changed (new/changed/removed endpoint, request/response shape).
- Editing `orval.config.ts`, the custom mutator (`orval-client.ts`), or the `build:api` script.
- Consuming an endpoint on the front end (a query/mutation hook, a response type).
- Reviewing a diff that hand-edits a generated file (`index.ts`, `*.schemas.ts`, per-tag files).

## Protocol

### Step 1 — Regenerate via the `build:api` pipeline

After any backend API change, regenerate — do not patch the client by hand.

```text
curl {BACKEND_URL}/docs-json > openapi.json   # pull the live spec
orval --config orval.config.ts                # generate the client
rm openapi.json                               # delete the temp spec
```

Run this as the project's `build:api` script. The generated output is disposable and fully reproducible from the spec.

### Step 2 — Never hand-edit generated files

```text
Need a generated file to behave differently?
  → change the SOURCE: the OpenAPI spec, orval.config.ts, or the mutator — then regenerate
  → NEVER edit index.ts / *.schemas.ts / per-endpoint files; the next build:api overwrites them
```

```text
BAD:  patch src/modules/mg-api/models.ts to fix a header → wiped on next regenerate
GOOD: fix the mutator or orval.config.ts, run build:api → fix persists
```

### Step 3 — `tags-split` output, one file per tag

Configure Orval with the `tags-split` output mode so each OpenAPI tag becomes its own file, not one monolith. Keep this in `orval.config.ts`; don't reorganize the generated tree manually.

### Step 4 — All requests go through the custom mutator

The custom fetch mutator (`src/utilities/orval-client.ts`) owns content-type dispatch (JSON / PDF / text) and wraps every response as `{ status, data, headers }`. Generated hooks call it; never bypass it with a raw `fetch`.

```ts
// BAD — raw fetch inside or alongside generated client; loses content-type dispatch + envelope
const res = await fetch(`${base}/models`)

// GOOD — the generated hook calls the mutator, which returns { status, data, headers }
const { data } = useGetModels()
```

### Step 5 — Derive types; use the generated hooks

- TypeScript types come from `z.infer<typeof Schema>` against the generated Zod schemas — never re-declare a parallel `interface` for an endpoint's shape.
- Use the generated React Query hooks (`useGetModels`, `useCreateTestDrive`); do not hand-write a `useQuery`/`useMutation` wrapper for an endpoint that already has a generated hook.

```text
Endpoint already has a generated hook?  YES → use it
                                        NO  → regenerate (Step 1); a missing hook means a stale client
```

## Completion checklist

- [ ] Client regenerated via `build:api` (curl spec → orval → rm temp), not hand-patched
- [ ] No edits to generated files; behavior changes made in spec / `orval.config.ts` / mutator
- [ ] Output stays `tags-split` (one file per tag)
- [ ] Every request flows through the custom mutator; no raw `fetch` for API calls
- [ ] Types via `z.infer<typeof Schema>`; no duplicated hand-written interfaces
- [ ] Generated React Query hooks used directly; no manual wrappers for generated endpoints

## When NOT to use

- The project has no Orval/OpenAPI setup, or the client is generated by a different tool (e.g. `openapi-typescript`, GraphQL codegen) — follow that tool's workflow.
- Calling an internal Next.js route handler or server action (no generated client involved).
