# payload-migrations

Use this skill whenever a schema-affecting change lands in a Payload CMS project — a new collection, a new/renamed/removed field, a changed column type — or whenever a file under `src/migrations/` is added, edited, or reviewed. The goal: the migration history stays a faithful, replayable record of the schema, generated by Payload rather than hand-authored, and destructive changes never run before the data they drop is confirmed dead.

## Triggers

- Adding or changing a collection, global, or field that alters the database schema (new column, changed type, dropped field).
- Any new or edited file under `src/migrations/`.
- A teammate hand-wrote migration SQL, or edited `payload-generated-schema.ts` / `payload-types.ts` directly.
- Reviewing a PR whose diff touches `src/migrations/` or the generated schema/types files.

## Protocol

### Step 1 — Generate the migration; never hand-write it

Run Payload's generator so the `up`/`down` SQL is derived from the schema diff, not guessed.

```text
Schema changed and a migration is needed?
  YES → run `payload migrate:create <descriptive-name>` (or the project's `generate:migration` script)
  NO  → stop; this skill does not apply
```

```text
BAD:  hand-authored src/migrations/20260617_add_locale.ts with a guessed ALTER TABLE,
      and an out-of-date snapshot (or no snapshot at all)
GOOD: pnpm generate:migration   # produces 20260617_120000.ts + 20260617_120000.json together
```

### Step 2 — Commit BOTH the `.ts` and the `.json` snapshot

Every generated migration is a pair: the executable `.ts` and a `.json` schema snapshot Payload uses to compute the next diff. Committing one without the other corrupts future generation.

- Stage and commit `src/migrations/<name>.ts` **and** `src/migrations/<name>.json` in the same commit.
- Never edit the `.json` snapshot by hand.

### Step 3 — Keep generated files generated

`payload-generated-schema.ts` and `payload-types.ts` are outputs, not sources. Editing them creates drift that the next regenerate silently reverts.

```text
Need to change types or the generated schema?
  → change the collection/field config, then regenerate: `pnpm generate:all`
  → NEVER edit payload-generated-schema.ts or payload-types.ts directly
```

### Step 4 — Require `up` and `down`, run raw SQL

Each migration MUST export both `up(args: MigrateUpArgs)` and `down(args: MigrateDownArgs)` so it can roll forward and back. Migrations run raw SQL through `db.execute(sql\`...\`)` — do not introduce ORM/query-builder abstractions inside a migration.

```ts
import { MigrateUpArgs, MigrateDownArgs, sql } from '@payloadcms/db-postgres'

export async function up({ db }: MigrateUpArgs): Promise<void> {
  await db.execute(sql`ALTER TABLE "models" ADD COLUMN "subtitle" varchar;`)
}

export async function down({ db }: MigrateDownArgs): Promise<void> {
  await db.execute(sql`ALTER TABLE "models" DROP COLUMN "subtitle";`)
}
```

### Step 5 — Gate destructive operations

`DROP COLUMN` / `DROP TABLE` / type narrowing destroy data irreversibly in `up`.

```text
Does the migration drop a column/table or narrow a type?
  YES → confirm the data is no longer read anywhere, then land the destructive step
        as a SEPARATE migration AFTER the deploy that stopped using it
        (expand → migrate-data → contract). Make `down` recreate the structure.
  NO  → proceed normally
```

## Completion checklist

- [ ] Migration produced by `payload migrate:create` / `generate:migration`, not hand-written
- [ ] Both the `.ts` migration and its `.json` snapshot are staged in the same commit
- [ ] No manual edits to `payload-generated-schema.ts` or `payload-types.ts`; types regenerated via `generate:all`
- [ ] `up` and `down` both exported; SQL run via `db.execute(sql\`...\`)`
- [ ] Any destructive op is isolated in its own post-deploy migration with the data confirmed unused

## When NOT to use

- The change is admin-UI-only (labels, field order, access rules) and produces no schema diff.
- A non-Payload migration tool (Prisma Migrate, Drizzle Kit, raw Flyway) owns the schema — follow that tool's workflow instead.
